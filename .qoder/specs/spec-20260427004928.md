# aoaojiao-rpc 优化实施方案

## 背景

根据 `docs/optimization-review.md` 的分析，项目存在多个优先级的问题需要优化：
1. **高优先级**：连接管理、资源泄漏、方法调用
2. **中优先级**：重试策略、负载均衡、限流器、熔断器
3. **低优先级**：序列化扩展、安全性

本方案专注于实施 8 个核心优化点。

---

## 一、高优先级优化

### 1.1 连接池实现（🔴 P0）

**问题**：
- `SimpleRpcClient` 每次调用都创建新连接
- `FailRetry` 重试时新建连接，开销翻倍
- `RpcClient` 无连接池机制

**修改文件**：
- `rpc-core/src/main/java/com/aoaojiao/rpc/core/pool/` (新增)

**实现方案**：
1. 新增 `PoolConfig.java` - 连接池配置类
2. 新增 `RpcConnection.java` - 连接接口
3. 新增 `ReusableRpcConnection.java` - 可复用连接实现
4. 新增 `ConnectionPoolManager.java` - 连接池管理器接口
5. 新增 `PerNodeConnectionPool.java` - 单节点连接池
6. 改造 `RpcClient.java` - 支持连接复用
7. 改造 `FailRetry.java` - 使用连接池而非每次新建

**关键设计**：
- 共享 EventLoopGroup，减少线程创建开销
- 引用计数管理连接生命周期
- 支持心跳检测和自动重连

### 1.2 PendingRequests 超时清理（🔴 P0）

**问题**：超时的请求会一直留在 pending map 中，导致内存泄漏

**修改文件**：
- `rpc-core/src/main/java/com/aoaojiao/rpc/core/transport/PendingRequests.java`

**实现方案**：
1. 添加 `ScheduledExecutorService` 定时扫描超时的 pending 请求
2. 为每个请求记录 deadline，清理时调用 fail()
3. 新增 `TimeoutListener` 回调机制用于监控/重试

**关键设计**：
- 清理间隔可配置（默认 5s）
- 请求超时后可触发重试回调

### 1.3 方法调用优化（🔴 P0）

**问题**：`DefaultServiceInvoker` 使用 `getMethod()` 而非 `getDeclaredMethod()`，可能匹配到父类方法

**修改文件**：
- `rpc-server/src/main/java/com/aoaojiao/rpc/server/invoker/DefaultServiceInvoker.java`

**实现方案**：
1. 新增 `MethodCache.java` - 方法缓存类，按 `className#methodName#paramTypes` 缓存 Method 对象
2. 改造 `DefaultServiceInvoker` - 使用方法缓存，调用 `getDeclaredMethod()` + `setAccessible(true)`

**关键设计**：
- 方法缓存减少反射查找开销
- 确保精确匹配当前类的方法

## 二、中优先级优化

### 2.1 负载均衡器修复（🟡 P1）

**问题**：`RoundRobinLoadBalancer` 的 `Math.abs(Integer.MIN_VALUE)` 返回负数，导致 `ArrayIndexOutOfBoundsException`

**修改文件**：
- `rpc-client/src/main/java/com/aoaojiao/rpc/client/cluster/impl/RoundRobinLoadBalancer.java`

**实现方案**：
- 使用 `AtomicLong` 替代 `AtomicInteger`
- 使用位运算安全取模：`index.getAndIncrement() & 0x7FFFFFFF`

### 2.2 重试策略优化（🟡 P1）

**问题**：
- FailRetry 重试时重复选择同一节点
- 重试间隔无延迟，可能雪崩
- 未区分可重试/不可重试异常

**修改文件**：
- `rpc-client/src/main/java/com/aoaojiao/rpc/client/cluster/fault/FailRetry.java`

**实现方案**：
1. 新增 `SmartRetryFaultTolerance.java` - 智能重试实现
2. 实现失败节点排除机制（使用 `ConcurrentHashSet`）
3. 实现指数退避 + 抖动算法
4. 添加异常分类重试

**配置项**：
```yaml
retry:
  max-attempts: 3
  base-interval-ms: 100
  max-interval-ms: 5000
  jitter-ms: 50
```

### 2.3 限流器改进（🟡 P1）

**问题**：`FixedWindowRateLimiter` 存在临界突刺问题

**修改文件**：
- `rpc-client/src/main/java/com/aoaojiao/rpc/client/cluster/limit/FixedWindowRateLimiter.java`
- 新增 `SlidingWindowRateLimiter.java`
- 新增 `TokenBucketRateLimiter.java`

**实现方案**：
1. 保留 `FixedWindowRateLimiter` 用于向后兼容
2. 新增 `SlidingWindowRateLimiter` - 滑动窗口限流，消除临界突刺
3. 新增 `TokenBucketRateLimiter` - 令牌桶算法，支持突发流量

### 2.4 熔断器增强（🟡 P1）

**问题**：只支持计数模式，无法基于错误率动态调整

**修改文件**：
- `rpc-client/src/main/java/com/aoaojiao/rpc/client/cluster/circuit/CircuitBreaker.java`

**实现方案**：
1. 新增 `SlidingWindowStats.java` - 滑动窗口统计
2. 扩展 `CircuitBreaker` 支持失败率模式
3. 添加滑动窗口内的动态阈值调整

**模式支持**：
- `COUNTING`: 连续失败 N 次触发熔断
- `FAILURE_RATE`: 滑动窗口内失败率超过阈值触发熔断

## 五、验证方案

### 5.1 单元测试

```bash
# 运行所有单元测试
mvn test -DskipITs

# 运行指定模块测试
mvn test -pl rpc-core -DskipITs
mvn test -pl rpc-client -DskipITs
```

### 5.2 集成测试

```bash
# 启动测试环境
# 测试连接池
# 测试超时清理
# 测试重试策略

mvn verify -DskipITs=false
```

### 5.3 性能基准测试

```bash
# 运行 RPC 性能测试
mvn test -Dtest=RpcPerformanceTest
```

### 5.4 手动验证点

1. **连接池**：多次调用同一服务，确认连接复用（观察日志/指标）
2. **超时清理**：模拟超时请求，观察 pending map 大小变化
3. **重试策略**：触发失败重试，观察退避间隔
4. **限流器**：快速发送请求，观察限流行为

## 六、向后兼容性

1. 连接池默认关闭（`pool.enabled=false`），用户需要显式开启
2. 现有 `SimpleRpcClient`、`RpcClient` 接口保持不变
3. 新增 `SmartRetryFaultTolerance` 作为可选重试策略
4. 限流器/熔断器支持配置切换新旧实现

---

## 七、风险缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 连接池耗尽导致阻塞 | 高 | 配置 acquireTimeout；使用信号量限制 |
| 定时清理线程竞争 | 中 | 使用 ConcurrentHashMap；读写分离 |
| 向后兼容问题 | 低 | 保持现有接口不变；新功能默认关闭 |
| 测试覆盖不足 | 中 | 补充单元测试；手动验证关键流程 |
